# ADR-017: Knowledge-graph self-reflection practice (Graphify + Codegraph)

- **Status:** Accepted
- **Date:** 2026-07-01
- **Deciders:** Maintainer
- **Related:** ADR-011 (OpenAPI auto-generation), `docs/Graphify-Integration-Guide.md`, `.mcp.json`, `.github/workflows/graphify.yml`

## Context

Nexus-HEMS-Dash is a large, safety-critical monorepo (13 protocol adapters, a dual
Zustand-store frontend, an Express backend, shared Zod contracts). Two complementary
knowledge-graph tools are wired into the repo:

- **Graphify** — an AST-derived knowledge graph committed under `graphify-out/`
  (`GRAPH_REPORT.md` for god-nodes/community structure, `graph.json`, `graph.html`),
  regenerated by `.github/workflows/graphify.yml` on pushes to `main` and available as
  an MCP server via `.mcp.json`. The regenerating AST **cache** (`graphify-out/cache/`)
  is git-ignored so it doesn't bloat history; final outputs are committed so the whole
  team shares one graph. Usage is documented in `docs/Graphify-Integration-Guide.md`.
- **Codegraph** — an optional per-project tree-sitter symbol/edge/call graph exposed
  through `codegraph_*` MCP tools (search, callers/callees, trace, impact, context).
  It answers *structural* questions (what calls what, blast radius, where-defined)
  sub-millisecond. This repo is **not** `codegraph init`-ed today; the tools work
  per-project against any directory that has a `.codegraph/` index.

These were previously ad-hoc developer aids with no recorded policy. Because the
project's working rules already mandate **truth-sync docs** and **safety-first change
review**, a standing decision about *how* the knowledge graph is used — and kept
honest — is worth recording so the practice survives contributor turnover.

## Decision

Adopt knowledge-graph consultation as a **standing self-reflection practice**, not a
one-off:

1. **Graphify outputs are committed, cache is not.** Keep `graphify-out/GRAPH_REPORT.md`,
   `graph.json`, `graph.html` in version control (shared team graph); keep
   `graphify-out/cache/` git-ignored (regenerated locally with `graphify update .`).
   This matches the existing Omni-Archive team policy and `.gitignore`.

2. **Consult the graph before architectural work.** For "how does X relate to Y",
   god-node/hotspot, control-path, or blast-radius questions, read
   `graphify-out/GRAPH_REPORT.md` (or query the graph / use `codegraph_*` when a
   `.codegraph/` index exists) **before** grep-and-read spelunking. Structural queries
   are what the graph is for; literal-text search stays with grep.

3. **Safety-critical use.** The graph is explicitly a tool for tracing **control paths**
   (sensor → controller → actuator), enumerating **danger commands** and their
   validation guardrails, and confirming adapter-interface contracts before touching
   the command pipeline — complementing (never replacing) the `command-safety.ts`
   review.

4. **Regeneration cadence.** The graph is regenerated automatically on every `main`
   push (`graphify.yml`); after a substantive local refactor, run `graphify update .`
   (AST-only, no API cost) so a stale graph never misleads a review. Treat the graph as
   *advisory*: it reflects the AST at generation time, so verify a named symbol/edge
   still exists before acting on an old report.

5. **Codegraph stays opt-in per project.** We do **not** commit a `.codegraph/` index
   (it is a local, regenerable cache). Initializing it is a contributor's choice
   (`codegraph init`); the MCP tools resolve the nearest index at query time.

## Consequences

- **Positive:** architectural and safety-path reasoning starts from a shared,
  AST-accurate map instead of divergent mental models; blast-radius checks before
  refactors are cheap; the practice is now discoverable and enforceable in review.
- **Positive:** no new runtime dependency or attack surface — both tools are
  build-time/developer-time only and never ship in the app bundle.
- **Negative / residual:** committed graph artifacts add churn to `main` on
  graph-affecting changes (bounded — cache excluded); a graph can go stale between
  regenerations, so it is advisory and must be re-verified, never trusted blindly for
  a safety decision.
- **Revisit if:** the committed artifacts prove noisy in review (move to artifact-only
  CI upload), or Codegraph is adopted repo-wide (then document the `.codegraph/`
  lifecycle and add an init step to `CONTRIBUTING.md`).
